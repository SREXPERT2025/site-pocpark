# ROSPARK Frontend — Content Guide (для контент‑менеджера)

Документ: `content-guide.md`  
Цель: самостоятельно добавлять и обновлять контент сайта без правок кода.

---

## 1) Где хранится контент

Весь контент находится в папке:

- `/content`

Внутри — разделы (папки):

- `/content/resheniya` — **Решения**
- `/content/oborudovanie` — **Оборудование**
- `/content/keysy` — **Кейсы**

Каждая страница — это **один Markdown-файл** (`.md`).  
Имя файла = **slug** (часть URL).

Пример:
- файл: `/content/resheniya/online-oplata.md`
- URL на сайте: `/resheniya/online-oplata`

---

## 2) Правила именования slug (имя файла)

Рекомендуемый формат:
- латиница
- нижний регистр
- дефисы вместо пробелов

Примеры:
- `online-oplata.md`
- `barrier-rv3.md`
- `case-tc-galereya.md`

---

## 3) Структура Markdown файла

Файл состоит из двух частей:

1) **Frontmatter** (в начале файла, между `---`) — это мета‑данные страницы  
2) **Markdown‑контент** — основной текст

---

## 4) Поля frontmatter по типам страниц

### 4.1. Решения (`/content/resheniya/*.md`)

Обязательные:
- `title` — заголовок страницы
- (рекомендуется) `description` — краткое описание для карточек и SEO

### 4.2. Оборудование (`/content/oborudovanie/*.md`)

Обязательные:
- `title`
- `description`

Дополнительные:
- `gallery` — массив путей к изображениям (строки)
- `specifications` — список характеристик (пары `name/value`)

### 4.3. Кейсы (`/content/keysy/*.md`)

Обязательные:
- `title`
- `description`

Дополнительные:
- `coverImage` — путь к обложке кейса
- `projectMetrics` — список метрик (пары `label/value`)

---

## 5) Где лежат изображения и как их добавлять

Изображения лежат в папке:

- `/public/images/...`

> Все файлы из `/public` доступны по URL, совпадающему с путём.

Пример:
- файл: `/public/images/cases/galereya-cover.jpg`
- в Markdown / frontmatter: `"/images/cases/galereya-cover.jpg"`

Рекомендации:
- Формат: `.webp` (предпочтительно) или `.jpg`
- Размер: 1600–2000px по ширине для обложек, 1200–1600px для карточек/галерей
- Вес: стараться держать < 300–500 KB (для WebP)

## 7) Публикация изменений

Важно: контент Markdown попадает на сайт после **сборки** проекта.

Если у вас доступ к репозиторию:
1) Создайте/отредактируйте `.md` и изображения
2) Сделайте commit/push
3) DevOps выполняет деплой (`npm run build` → restart)

Если вы работаете через файловый доступ на сервер:
1) Добавьте `.md` в `/content/...`
2) Добавьте изображения в `/public/images/...`
3) DevOps пересобирает проект

---

## 8) Частые ошибки

- **Забыли `title`** → страница может не появиться в списках.
- **Неверный путь к картинке** → будет 404. Убедитесь, что путь начинается с `/images/...` и файл реально лежит в `/public/images/...`.
- **Слишком тяжёлые картинки** → ухудшение Lighthouse. Используйте WebP и компрессию.
- **Дубликат slug** → один файл перезапишет другой по смыслу. Slug должен быть уникальным.

---

## 9) Быстрый чек перед передачей на прод

- Заголовок (`title`) есть
- `description` заполнено (желательно)
- Изображения открываются по URL
- Текст проверен на орфографию
- Есть минимум 1–2 подзаголовка (`##`), списки, чёткая структура
